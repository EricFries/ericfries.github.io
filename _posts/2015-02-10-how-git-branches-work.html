--- layout: post title: How Git Branches Work ---

Git branches are amazing tools that allow developers to test new features without wreaking havoc on code that already works. When combined with GitHub, branches are especially useful to developers who need to merge their work with collaborators.

As a new developer, I'm frequently in the mindset of following established rules and trusting that certain techniques and concepts just work. I don't need to know every detail about how something works, but I do get a little freaked out when I have no idea how something is working. And until now, I had no idea how Git branches actually worked.

This is where things get a little mind-bending. How is git able to do this? You might be thinking that if I opened my Finder, and opened the directory containing the textfile, that there might actually be two textfiles. You may even be thinking that git secretly created a "master" branch directory and an "experimental-theories" branch directory, and that each exists as a parallel universe with their own version of the textfile I've been modifying. At least, that's what I thought, and it's wrong.

There is exactly one post.txt file in the directory I've saved it in. But this is troubling becuase when I swtich back and forth between branches I'm seeing different verisons of post.txt. This is the magic of git. Remember those snapshots of post.txt I took using "git commit?" Everytime I switch the branch I'm working in, git relies on these snapshots to modify the one, singular file to look exactly as it did according to the last snapshot taken in the branch I'm switching to.
